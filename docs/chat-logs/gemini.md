제공된 문서에 따른 **개인화 RAG 시스템 구축을 위한 리서치 결과**는 다음과 같습니다. 1인 개발자가 유지보수 부담을 줄이면서도 확장성을 확보할 수 있는 '최적의 기술 스택(One Pick)'과 그 선정 근거가 핵심 결과입니다.

### 1. 최적의 아키텍처 선정 결과 (The One Pick)

리서치 결과, 가성비와 유지보수 용이성을 모두 만족하는 1인 개발자용 최적의 조합은 **Obsidian + Supabase + n8n + Dify**로 도출되었습니다.

* **소스(Source):** Obsidian (로컬 마크다운 노트).
* **저장소(Memory):** Supabase (Vector DB & PostgreSQL).
* **연결(ETL):** n8n (워크플로우 자동화).
* **두뇌(Brain):** Dify (RAG 엔진 및 오케스트레이션).
* **인터페이스(Interface):** Claude Desktop (MCP 활용).

### 2. 도구별 비교 분석 결과

각 구성 요소별로 여러 툴을 비교 분석한 결과는 다음과 같습니다.

* **벡터 데이터베이스 (Vector DB): Supabase 선정**
  * **결과:** Pinecone은 편의성이 높으나 데이터가 많아질수록 비용이 급증하며, Weaviate는 성능은 좋으나 직접 호스팅하고 관리하기가 까다롭습니다.
  * **선정 이유:** Supabase는 월 \$25 정도의 합리적인 비용(Pro 요금제)으로 이용 가능하며, 관계형 DB(PostgreSQL)와 벡터 검색(`pgvector`)을 하나의 플랫폼에서 관리할 수 있어 1인 개발자에게 가장 적합합니다.
* **ETL 파이프라인 (Data Pipeline): n8n 선정**
  * **결과:** LangChain은 코드를 직접 짜야 해서 유지보수 피로도가 높고, Airbyte는 설정이 복잡하고 무겁습니다.
  * **선정 이유:** n8n은 노드 기반의 시각적 인터페이스를 제공하여 직관적으로 파이프라인을 구축할 수 있으며, Obsidian 플러그인 등 다양한 연동을 지원해 자동화가 용이합니다.
* **RAG 엔진 (Orchestration): Dify 선정**
  * **결과:** Flowise는 UI가 직관적이지만 기능 확장이 제한적이고, LangChain은 강력하지만 러닝 커브가 높습니다.
  * **선정 이유:** Dify는 직관적인 GUI를 제공하면서도 백엔드 API를 자동으로 생성해주어 확장이 쉽습니다. 또한, 다양한 LLM 모델(GPT, Claude 등)을 손쉽게 교체할 수 있는 유연성을 가집니다.

### 3. 핵심 기술 차별점: MCP (Model Context Protocol) 도입

단순한 챗봇을 넘어, AI가 사용자의 로컬 데이터와 도구를 직접 제어할 수 있는 **MCP(Model Context Protocol)** 기술을 도입하는 것이 미래지향적인 결과로 제시되었습니다.

* **효과:** Dify로 구축한 RAG API를 MCP 서버로 래핑하여 Claude Desktop과 연결하면, AI가 내 Obsidian 노트를 실시간으로 참고하여 답변하는 '진정한 두뇌 확장'이 가능해집니다.
